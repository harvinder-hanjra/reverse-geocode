# h3 — H3 cell binary search geocoder

Binary format: **LKHA0001**
Builder: Python (`builder.py`)
UI: `ui/` (port 5176)

## How it works

The simplest of the three implementations: a flat sorted array of raw 64-bit
H3 cell IDs, searched with a standard binary search.

### Index structure

For each ADM2 region, the builder fills all H3 cells at **res-5** (coarse,
~252 km² average) that fall inside the polygon, then fills cells at **res-6**
(~36 km²) for cells that straddle a border. Interior cells use the coarser
resolution to keep the record count manageable; boundary cells use the finer
resolution for accuracy.

Each record is **12 bytes**:

```
uint64  h3_index    raw H3 cell ID (little-endian)
uint32  packed_meta country_id[7:0] | state_offset[15:8] | district_offset[31:16]
```

The array is sorted by `h3_index`, enabling binary search.

### Lookup algorithm

Given (lat, lon):
1. Compute H3 res-6 cell → binary-search the array.
   If found, decode `packed_meta` → return `{country, state, district}`.
2. Compute res-5 parent → binary-search again.
   If found, return result.
3. Compute res-4 parent → binary-search again.
   If found, return result.
4. No match → return `null` (ocean or unmapped).

The three-resolution fallback ensures that even very small polygons (which
may not contain any res-6 cell centroid) are eventually covered by a coarser
parent cell.

### Name table

Names are stored as **nested lists** keyed by `country_id`:

```json
{
  "countries": ["UNK", …],
  "adm1": [["state0", "state1", …], …],
  "adm2": [["district0", "district1", …], …]
}
```

`state_offset` and `district_offset` index into the per-country sublists.
This structure caps both fields at 256 and 65 536 entries per country
respectively.

The table is zstd-compressed inside the binary. For browser use a
pre-extracted `h3_names.json` is generated by `extract_names.py` to avoid
shipping a WASM decompressor.

### Binary format (LKHA0001)

```
[0:8]    magic "LKHA0001"
[8:12]   version       uint32
[12:16]  n_records     uint32
[16:20]  name_table_off uint32
[20]     record array  (n_records × 12 bytes, sorted by h3_index)
[…]      name table    (zstd JSON)
```

### Building

```sh
cd h3
pip install -r requirements.txt
python builder.py <gadm_adm2.geojson>
# produces h3_geo.bin

python extract_names.py   # produces h3_names.json for browser use
```

### 64-bit keys in JavaScript

H3 cell IDs exceed JavaScript's safe integer range. The browser runtime
(`ui/src/h3.js`) uses `BigInt` for all comparisons:

```js
const lo32 = dv.getUint32(off,     true);
const hi32 = dv.getUint32(off + 4, true);
const key  = (BigInt(hi32) << 32n) | BigInt(lo32);
```

### Performance

- Typical query: ~2–5 µs (BigInt arithmetic adds overhead vs. 32-bit keys)
- File size for geoBoundaries ADM2: ~10 MB
